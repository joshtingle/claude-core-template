# Database: PostgreSQL

Covers Postgres on Supabase, AWS RDS, and self-hosted.  Variant-specific notes called out below.

## Connection

Required env vars:

- `PG_HOST`
- `PG_PORT` (default 5432)
- `PG_DATABASE`
- `PG_USERNAME`
- `PG_PASSWORD`
- `PG_SSL_MODE` (`require` for Supabase, `verify-full` for production RDS)

For Supabase specifically, also capture:

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY` (client-side, safe to expose in frontend)
- `SUPABASE_SERVICE_ROLE_KEY` (server-side only, NEVER expose to client)

## SDKs

- Node: `pg` (node-postgres) for direct access, `@supabase/supabase-js` for Supabase client
- Python: `psycopg` v3
- .NET: `Npgsql`

## Variant: Supabase

Supabase provides Postgres plus REST/GraphQL API, auth, storage, and realtime.  When the project uses Supabase, prefer the Supabase client library (`@supabase/supabase-js`) for app-side reads and writes, and reserve direct `pg` access for server-side jobs and migrations.

Row-Level Security (RLS) is on by default for tables created through the dashboard.  When creating tables via migration, explicitly enable RLS and write policies in the same migration.  A table without RLS policies is invisible to the API even if data is in it.

The `pgvector` extension is enabled by default and is the right choice for projects that need semantic search (StoreSmart-style).

## Variant: AWS RDS Postgres

Same network and security group considerations as RDS SQL Server.  See `sqlserver-rds.md` for the network setup pattern.  Substitute port 5432 for 1433.

## Query patterns

Use `RETURNING` clauses on INSERT/UPDATE/DELETE to get back affected rows in one round trip.

Use `ON CONFLICT (column) DO UPDATE SET ...` for upserts.  Don't write SQL Server-style explicit branch.

For pagination, prefer keyset pagination (WHERE clause references the last seen value) over OFFSET for large tables.  OFFSET scans all skipped rows.

JSONB is the right choice for semi-structured data, but don't reach for it as a substitute for proper schema design.  If a JSONB field is being indexed and queried on specific keys, those keys probably want to be columns.

## Schema migrations

Use a migration tool: `node-pg-migrate` for Node, `alembic` for Python, EF Core migrations for .NET, or hand-rolled sequential SQL files in `db/migrations/`.  Whichever pattern, never alter schema interactively in committed flow.

## SQL style

```sql
-- File: db/migrations/003_create_inventory.sql
-- Purpose: Inventory items table for StoreSmart, with pgvector embedding column.
-- Last updated: 2026-05-23 (Claude Code)

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE IF NOT EXISTS inventory_item (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    qr_code TEXT UNIQUE,
    name TEXT NOT NULL,
    description TEXT,
    embedding VECTOR(1536),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX inventory_item_user_idx ON inventory_item(user_id);
CREATE INDEX inventory_item_embedding_idx ON inventory_item USING ivfflat (embedding vector_cosine_ops);

ALTER TABLE inventory_item ENABLE ROW LEVEL SECURITY;

CREATE POLICY "users see their own items"
    ON inventory_item FOR SELECT
    USING (auth.uid() = user_id);

CREATE POLICY "users manage their own items"
    ON inventory_item FOR ALL
    USING (auth.uid() = user_id);
```

Lowercase keywords are common in the Postgres community.  Either style is fine, but be consistent within a project.

## Things to never do

Don't disable RLS on Supabase tables to "fix" an access issue.  Write the policy correctly.

Don't store dates as `TEXT` or `INT` (epoch).  Use `TIMESTAMPTZ` or `DATE`.

Don't use `SERIAL` for new tables.  Use `GENERATED BY DEFAULT AS IDENTITY` or UUID primary keys.

Don't connect to Supabase from a client app using the service role key.  That key bypasses RLS and exposes the entire database if leaked.
